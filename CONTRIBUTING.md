# Contributing

Thanks for taking the time to contribute to OrxaCode.

We are open to pull requests, but we are still shaping the product, so we are selective about what we merge. The best contributions are focused, well explained, and clearly improve the app without pushing it in a new direction we have not agreed on.

## What We Are Most Likely To Accept

- Bug fixes
- Reliability improvements
- Missing features that feel like obvious gaps in the current product
- Plugin, extension, or integration support that fits the direction of the app
- UX and UI polish
- Personalization work such as themes or appearance options
- Small performance or accessibility improvements
- Documentation that makes the project easier to use or contribute to

## What Usually Needs Discussion First

Please open an issue before starting work on:

- Large features
- Architectural rewrites
- Changes that add a new product direction
- Big dependency swaps
- Anything that changes core workflows or app behavior in a major way

That does not mean we are against bigger contributions. It just means we want to agree on the shape first so nobody wastes time.

## Before You Open a PR

Please:

1. Fork the repo
2. Create your branch from `dev`
3. Keep the change focused
4. Run the relevant checks locally
5. Open the PR against `dev`

If your change is non-trivial, link the issue or discussion that explains the context.

## What We Expect In a Pull Request

Please explain your change properly. A good PR should make it easy for us to understand:

- what changed
- why it changed
- how it was tested
- any tradeoffs or limitations

If the PR touches the UI, include before/after screenshots.

If the change depends on motion, interaction, or timing, include a short video or GIF.

If the PR is large, please explain why it had to be large.

## Good PRs Tend To Be

- Small to medium in size
- Easy to review in one pass
- Limited to one problem or one feature
- Consistent with the existing product direction

## PRs We Are Less Likely To Merge

We are less likely to take:

- Drive-by rewrites
- Broad refactors with no clear user benefit
- Opinionated changes that replace existing product decisions without discussion
- Mixed PRs that bundle unrelated fixes together
- Large changes with little explanation

## A Few Practical Notes

- We do not use a vouch system for contributors at the moment
- Opening a PR does not guarantee it will be merged
- We may ask you to reduce scope, split the work, or move the discussion to an issue first
- We may close a PR that is not a fit, even if the work is well intentioned

That said, well-scoped contributions are welcome, and we appreciate the effort.

## If You Are Unsure

Open an issue first and outline the change you want to make.

That is the fastest way to find out whether the work is likely to be accepted before you spend time building it.
